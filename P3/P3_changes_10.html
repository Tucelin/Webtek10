Changes

This section details the changes that have been made to the final website compared to the original design laid out in the design document.

Navigation structure

We did not make a splash screen. This was an interesting idea that we included in the original design, but because we prioritized refining the design and functionality of the more important pages of the site.



Page layout and appearance

Due to our understanding of the modularity requirement, we had to rework our navbar so that it would be modular, and in doing so, we lost a bit of intended functionality. We intended for the page the user is currently visiting to be lit up on the navbar to clearly indicate the users whereabouts, but this functionality was hard to implement in our new, general, modular solution. We ended up putting a page indicator at the bottom of the page in the footer.

For the navbar, we added some empty space on each side for larger screens. This looks better than dragging it all the way out to the sides. We did not prioritize implementing that the navbar hides when scrolling down the page. This is not so important because most of the pages are not scrollable on the most common screen sizes. The font sizes are not the same that were specified in the original design. The font sizes have been adjusted based on what we found to work best when experimenting during the implementation.



Home.html

The title “CleverNest” was added to the top of the content section make it more clear what site the user has arrived at.



Packages.html

In principle, there are no changes to the packages page. Because we were provided less text from the client than we expected, the overall look of the page is different than the mockup. For the same reason, we have not included a separate text about the basic package.



Providers.html

The input field for the postal code was moved above the array of provider logos. We decided it makes more sense to put the widget the user will interact with near the top of the page, as it is the central feature of the page.



Prices.html

There was some inconsistency between the textual description and the mockup for this page in the design. The final implementation is in line with the textual description. We also added some explanatory text below the price table because we found the user would need some more explanation to understand the price table better.



About.html

The client requested that we not include information about employees. Because the company is in a start-up phase, some details like address are not decided yet and are not included.



Support.html

The visual design of the download links and contact form was tweaked and improved during the implementation process. The download functionality is implemented as a regular HTML-link instead of through Javascript. With so few documents available for download,

Modularity requirement

Modularity was hard to implement in a simple way. We ended up injecting the code using JavaScript with the id-selector to get the navbar modular. The html-code used for the the navbar before we switched to modular is still included in the HTML files as comments. The original solution included a lighter color around the current page in the navbar to indicate what page the user is on. This functionality was lost with the modular implementation (at least when using the techniques we have learned in this course). Vegard Ekra translated the existing navbar into the modular solution we’re delivering. Because part of the navbar is created using Javascript, we were not able to use the same method to modularise it.
We used the same CSS-file for the entire project, as advised in the assignment text.
The only Javascript function used across several pages is copyrightYear.js. It is defined in one common file for all pages.
